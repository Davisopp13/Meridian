Rev. 2 · April 17, 2026 Alpha day 3 · data flowing

Where we are

Mid-alpha. Training wrapped April 8. Real data capture started April 14 and has now had three days in production with ~8 agents across CH and MH. The work below is deliberately post-training and pre-leadership-review: it assumes the widgets are stable, the bookmarklet flow is proven, and the priority is now turning captured data into something supervisors can defend to their bosses, while simultaneously opening a real feedback channel so the next two weeks of alpha observations don't get lost in Slack.

The ordering respects the three constraints that define mid-alpha: don't break live data capture, don't ship anything that can't be rolled back in ten minutes, and ship the feedback mechanism before the feature work so agents feel heard while longer builds (Insights, CT 1.0 sync) are in flight.

1

Current state

alpha day 3 · HEAD 3a3d53b

The widget bifurcation is complete — CT at ?mode=ct-widget, MPL at ?mode=mpl-widget — and both have been in agents' hands for three days. The original combined widget (src/App.jsx, 1,627 lines) still carries the dashboard shell and significant dead code. No role hierarchy, no teams table, no supervisor surface, no feedback channel.

Audit finding (April 17): RFC in the code is a correction event — fired only when an agent resolves a case that was previously resolved by someone else. The excluded column on case_events is written but never read; it can be ignored or dropped later. The real metric hiding underneath is how many times each case has been resolved. Track 2b (new) adds a reopen_count column to ct_cases, stamped at write time, so Insights can surface chronic cases as their own first-class panel.

What's working

  • Self-serve onboarding · bookmarklet install · auth
  • CT widget: case logging, stat buttons, RFC detection
  • MPL widget: process timer, category drill-down, manual entry
  • Dashboard: single-user stats, activity log, settings
  • Build clean: 814 KB, zero warnings

What's missing

  • No teams table · no role column
  • No InsightsTab · no supervisor-scoped queries
  • No feedback channel · no suggestion triage
  • No CT 1.0 historical backfill
  • App.jsx is the entry point AND carries dead widget code

The Suggestion Box

Ralph · 5 phases, 18 tasks

In-app feedback — bug reports, feature requests, new MPL category/subcategory proposals, open-ended feedback. Status lifecycle (new → acknowledged → in_progress → shipped / wont_fix), admin notes, and a one-click "Promote to live categories" action that atomically inserts approved category suggestions into mpl_categories and marks the suggestion shipped.

  1. Phase 1 — Schema: suggestions table, suggestion_attachments table, RLS, promote_suggestion(uuid) RPC, private Storage bucket

  5. Phase 5 (optional) — Supabase Edge Function + database trigger for email notifications on new suggestions

Decisions locked in

don't re-litigate

Structural

  • Delete App.jsx entirely — no "legacy fallback" hedge.
  • Four roles: agent, supervisor, director, admin.
  • Haulage type lives on the team, not the user.
  • Supervisor scope is via assignment (supervisor_teams), not role.
  • Davis starts as admin, assigned both Export Rail teams.
  • Keep old platform_users.team column for one deploy — drop later.

Suggestion Box

  • Admin access is a real role, not an email whitelist.
  • Five types · full status lifecycle · private admin notes.
  • Feedback tab visible to everyone; Admin tab gated.
  • Category promotion is a SECURITY DEFINER RPC with in-function role check.
  • Attachments: bugs only · one image · 5 MB cap · client-compressed.
  • Email notifications: Phase 5, opt-in via progress.txt.
