- `src/mpl/MplApp.jsx` — integrated all hooks, recovery detection on mount, beforeunload flush
- `src/mpl/MplPipBar.jsx` — renders `<RecoveryPrompt />` when `recoveredProcesses.length > 0`

### Storage Keys Introduced
- `meridian.mpl.active.v1` — localStorage key; JSON payload: `{ userId, savedAt, processes: [{ id, elapsed, paused, categoryId, subcategoryId, startedAt }] }`
- Stale threshold: 24 hours (STALE_THRESHOLD_MS)
- Crash threshold: snapshot older than 1 hour triggers recovery prompt

### New Supabase Columns (migration 009)
**`mpl_active_timers`**:
- `process_id` text — UUID from `crypto.randomUUID()`, used as upsert conflict key
- `subcategory_id` uuid nullable
- `accumulated_seconds` integer default 0
- `updated_at` timestamptz default now()

**`bar_sessions`**:
- `last_seen_at` timestamptz default now()
- `widget_mode` text (values: `'mpl-widget'` | `'ct-widget'` | `'legacy'`)

**Indexes added**:
- `idx_mpl_active_timers_user` on `mpl_active_timers(user_id, updated_at DESC)`
- `idx_bar_sessions_user_mode` unique on `bar_sessions(user_id, widget_mode)` (required for upsert onConflict)

### How to Manually Trigger Recovery for Testing
1. Open the MPL widget in Chrome DevTools
2. Start an MPL timer (any category)
3. Verify localStorage: `localStorage.getItem('meridian.mpl.active.v1')` — should show process with elapsed time
4. Kill the PiP window via DevTools → Application tab → Storage → Clear Session Storage (do NOT use graceful X; instead force-kill via Task Manager or `document.pictureInPictureElement.close()` from console without letting beforeunload fire cleanly)
5. Alternatively: simulate crash by navigating to Application → Local Storage, edit `savedAt` to a recent timestamp, then reload the page — recovery prompt should appear immediately
6. On reopen, `<RecoveryPrompt />` renders with accumulated time and category name
7. Test all three buttons: Resume (restores timer), Log now (calls logMplEntry), Discard (clears storage)

### Known Limitations
- **Windows Update / OS force-kill**: `beforeunload` does NOT fire on OS-level kill (e.g., Windows Update forcing browser shutdown). In this case, only the localStorage snapshot (written every tick) is the safety net. The Supabase `mpl_active_timers` row may be up to 10 seconds stale, but localStorage is authoritative and should be < 1 second stale.
- **sendBeacon on graceful close**: fires for tab close, window close, browser quit via File → Exit. Does not fire for process kill (`kill -9`, Task Manager end process).
- **No cross-device sync UI**: if the same user opens Meridian on a second machine while a session is active on another, recovery prompt will show on whichever they open next. This is acceptable behavior per PRD scope.
